# Crayotter Desktop GUI / Agent Workbench 方案文档

## 1. 文档目的

这份文档不直接进入编码，而是先把 Crayotter 从“命令行视频 Agent”升级为“面向小白用户的桌面工作台”的产品和技术方案讲清楚。

重点回答下面几个问题：

- 我们到底要做一个什么产品
- 为什么这个方向适合当前仓库
- 小白用户怎样做到尽量一键可用
- UI 应该长什么样，Agent 交互应该怎么呈现
- 底层技术怎么拆，哪些地方必须先重构
- 应该分几期做，先做什么，后做什么

---

## 2. 先说结论

我对这个项目的建议不是“再做一个宣传网站”，而是做一个真正可操作的桌面端工作台，形态上接近 qoder、codex、cursor 这类“任务驱动 + 流式过程可见 + 产物可回看”的客户端。

但 Crayotter 和普通代码 Agent 不一样，它的核心是“视频生产流水线”，所以桌面端不应该只像聊天窗口，而应该是：

- 左侧是任务与素材
- 中间是对话与 Agent 执行过程
- 右侧是计划、工具动作、产物、日志和视频预览

产品目标不是让用户学会命令行，而是让用户：

1. 第一次打开应用时填入 API Key / Base URL / 模型配置
2. 输入一句自然语言需求
3. 可选上传本地素材
4. 点击开始
5. 看见 Agent 自己规划、搜索、下载、分析、剪辑、配音、导出
6. 在过程中随时知道当前做到哪一步、为什么这样做、出了什么问题
7. 最后直接预览和导出视频

一句话概括：

**把现在的 `script/agent.py + script/graph.py + script/visualize.py`，产品化成一个可安装、可配置、可流式观察、可直接产出视频的桌面应用。**

## 3. 基于当前仓库的现状判断

从现有代码看，Crayotter 已经具备了桌面端产品的核心雏形，只是现在这些能力还散落在命令行、日志和脚本里。

### 3.1 已经有的能力

- 有明确的三阶段工作流
  - Phase 1: Planner + Executor
  - Phase 2: Editing Research
  - Phase 3: ReAct Editor
- 有状态模型
  - `script/graph.py` 里已经有 `AgentState`、`Plan`、`Step`
- 有工具系统
  - 搜索、下载、分析、剪辑、转场、配音、字幕、导出已经都在 `script/tools/` 中
- 有日志和轨迹
  - `script/visualize.py` 已经能把执行日志可视化
- 有工作目录约定
  - `temp/`、`user_temp/`、`logs/`、`memory_experience/`

### 3.2 现在不适合小白用户的地方

- API 配置虽然支持环境变量，但整体体验仍然偏工程化
- 主入口仍是命令行，用户需要自己运行 Python
- 进度可见性主要依赖日志文件，不是产品级实时界面
- 当前“流式过程”只在终端打印，不是结构化事件流
- 素材、输出、日志、记忆文件的关系，普通用户很难理解
- 一键启动能力还没有落地，依赖安装也对新手不友好

### 3.3 这意味着什么

这不是一个“从零开始设计产品”的项目，而是一个“需要把已有引擎包装成产品”的项目。

这很好，因为真正难的部分不是画 UI，而是让 Agent 真能完成任务。当前仓库已经证明引擎雏形是成立的，所以桌面化是正确方向。

## 4. 产品定位

## 4.1 产品定义

Crayotter Desktop 是一个面向非技术用户的视频 Agent 工作台。用户通过自然语言描述需求，系统自动完成素材检索、规划、分析、剪辑、转场、配音、字幕和导出，并把整个过程透明展示出来。

## 4.2 目标用户

- 内容创作者
- 校园宣传/活动运营
- 中小团队市场人员
- 不会命令行、不会剪辑软件、但愿意尝试 AI 自动视频生产的人

## 4.3 产品卖点

- 不需要会写代码
- 不需要懂 FFmpeg / MoviePy / LangGraph
- 不需要理解整个工作流细节也能跑通任务
- 过程是可见的，不是黑盒
- 出问题时知道卡在哪，不会只看到“失败”

## 4.4 非目标

第一期不追求：

- 做成完整专业 NLE 剪辑软件
- 支持多人协作云端项目管理
- 做成在线 SaaS 平台
- 做成全平台极致轻量的商业级安装器

第一期重点只有一个：**让个人用户在本地机器上可以真正顺畅地跑起来。**

## 5. 我建议做成什么样的 UI

我建议整体采用“三栏工作台”布局，而不是单栏聊天窗口。

### 5.1 主界面布局

#### 左栏：任务与素材区

- 新建任务
- 历史任务列表
- 本地素材上传区
- 输出文件列表
- 任务模板入口

这部分解决“我现在在做哪个项目”和“我上传了什么素材”的问题。

#### 中栏：主对话与执行流

- 顶部是任务输入框
- 中间是 Agent 对话流
- 流式显示：
  - 当前阶段
  - 当前子任务
  - 思考摘要
  - 工具调用动作
  - 结果摘要
- 支持用户中断、继续、重新规划

这部分是整个产品的“灵魂区域”，体验上最接近 qoder/codex。

#### 右栏：结构化执行面板

- 阶段进度条
- 任务清单
- 当前工具调用参数
- 生成的中间产物
- 视频预览
- 错误提示与修复建议
- 调试日志入口

这部分解决“透明感”和“可控感”。

### 5.2 为什么不能只做聊天窗口

因为这个项目不是纯问答型 Agent，而是长链路生产型 Agent。

用户真正关心的不是一句最终回复，而是：

- 现在在哪个阶段
- 搜到了什么素材
- 哪些素材被选中了
- 剪了哪些片段
- 配音是否完成
- 最终导出了什么文件

所以必须同时给出：

- 对话式体验
- 看板式状态
- 产物式反馈

## 6. Agent 风格交互应该怎么设计

你提到“像 qoder、codex 那样，任务清单，然后自己自动跑，并且流式地呈现思考和 action”，我完全认同，但要做一点适配。

### 6.1 推荐展示的不是原始 CoT，而是“思考摘要”

建议 UI 中展示以下三类流式信息：

- `Thinking Summary`
  - 例如：“我正在判断目标视频时长，并估算需要多少素材”
  - 例如：“我已分析 3 个候选视频，准备进入剪辑蓝图生成”
- `Action`
  - 例如：调用了 `search_bilibili_video`
  - 例如：调用了 `download_bilibili_video`
  - 例如：调用了 `add_narration_segments`
- `Observation`
  - 例如：找到 38 条候选素材
  - 例如：已下载 6 个视频，平均时长 42 秒
  - 例如：转场完成，当前成片时长 97.2 秒

不要直接展示不可控、冗长、容易误导用户的底层原始推理文本。

### 6.2 为什么这样设计

- 更稳定
- 更容易做前端结构化渲染
- 更适合普通用户理解
- 更方便后续做“暂停后继续”
- 更适合保存为任务轨迹

### 6.3 Agent 交互应支持的操作

- 开始任务
- 暂停任务
- 取消任务
- 从某阶段重跑
- 仅重新生成剪辑蓝图
- 仅重新配音/字幕
- 查看完整日志

这样用户不会觉得“一失败就要从头来”。

## 7. 首次启动和“小白可用”方案

这个部分最关键。

如果这里没做好，再漂亮的 UI 都是空的。

### 7.1 目标状态

理想体验应该是：

1. 用户下载安装包
2. 双击打开
3. 第一次启动进入配置向导
4. 填入 API Key
5. 选择服务商模板或手动填写 Base URL / Model
6. 点“测试连接”
7. 成功后进入主界面
8. 输入任务直接开始

### 7.2 配置项设计

建议把配置从代码里彻底抽离，变成 GUI 可编辑配置。

建议配置分三层：

- `Provider Presets`
  - 阿里 DashScope
  - OpenAI 兼容接口
  - 自定义 OpenAI-Compatible API
- `Model Settings`
  - 文本模型
  - 视频分析模型
  - TTS 模型
- `Runtime Settings`
  - 工作目录
  - 最大并发
  - 默认导出目录
  - 日志保留策略

### 7.3 API 输入后就能用，具体怎么实现

建议这样做：

- 启动时检查本地配置文件是否存在
- 不存在就弹出 Setup Wizard
- 配置写入本地加密或半加密存储
- GUI 后端启动时将配置注入运行时，不要求用户改 Python 文件

推荐配置文件：

- `app_config.json`
- `profiles/default.json`

敏感字段建议使用：

- Windows Credential Manager
- 或 Electron/Tauri 的安全存储封装

第一期如果来不及，也至少做到：

- API Key 不写死在源码里
- 由 GUI 存储和读取
- 运行时注入环境变量

### 7.4 “一键启动”怎么做才现实

我建议分层理解“一键启动”。

#### 方案 A：开发阶段一键启动

适合内部自用或快速试验：

- App 启动时拉起本地 Python 后端
- 自动检查虚拟环境
- 自动检查依赖
- 缺失则提示安装

优点：

- 开发快
- 改动最小

缺点：

- 对小白仍然不够友好

#### 方案 B：正式桌面版一键启动

适合真正给外部用户：

- 桌面应用内置 Python Runtime
- 打包 FFmpeg
- 打包所需脚本
- 首次启动自动做依赖探测
- 用户只需要填 API

优点：

- 最接近真正傻瓜式

缺点：

- 打包复杂度显著上升
- 安装包会更大

#### 我的建议

先做：

- 桌面壳 + 本地 Python 服务 + GUI 配置注入

再做：

- 打包 Python Runtime 和 FFmpeg

也就是先解决“能用”，再解决“真傻瓜”。

## 8. 技术架构建议

我建议把项目拆成三层，而不是让桌面端直接调用现在的脚本。

### 8.1 架构分层

#### 第一层：Core Engine

也就是现有的 Agent 核心能力，保留 Python。

包含：

- `graph.py`
- `agent.py`
- `tools/`
- 日志、工作目录、记忆沉淀

但需要重构成“可被程序调用的服务层”，而不是只适合 CLI。

#### 第二层：Local App Service

这是连接 UI 和 Python Agent 的桥梁，建议新增一个本地服务层。

建议技术：

- Python + FastAPI
- WebSocket 或 SSE

负责：

- 接收 UI 的任务请求
- 启动 Agent Job
- 推送流式事件
- 管理任务状态
- 暴露文件、日志、预览接口

#### 第三层：Desktop UI

建议桌面应用使用：

- `Electron + React`

原因：

- 前端生态成熟
- 流式 UI 容易做
- 调用本地进程方便
- 打包 Windows 桌面应用经验多
- 后续做自动更新、日志上传、崩溃收集更顺手

### 8.2 为什么第一版更推荐 Electron，而不是直接 Tauri

Tauri 也很好，包体更小，但当前项目要同时处理：

- Python 子进程
- 本地文件系统
- 视频预览
- 长任务流式通信
- 安装包兼容性

第一版优先级是稳定和开发速度，Electron 更务实。等桌面版跑稳后，再评估是否迁移到 Tauri。

### 8.3 服务调用关系

建议流程：

1. Electron 启动
2. Electron 拉起本地 Python 服务
3. Python 服务读取本地配置
4. UI 发起任务
5. Python 服务调用 Agent Runtime
6. Agent Runtime 持续产生结构化事件
7. 服务通过 WebSocket/SSE 推给 UI
8. UI 实时渲染阶段、动作、结果和预览

## 9. 为什么现有代码必须先做服务化重构

如果直接让前端执行 `python script/agent.py`，短期看似能跑，但很快会遇到问题：

- 事件流不结构化
- 很难暂停/取消
- 很难做任务历史管理
- 很难同时跑多个任务
- 错误只能从 stdout 里硬解析
- 配置注入很乱

所以必须把执行入口改成“任务服务”。

### 9.1 需要新增的核心模块

建议新增：

- `app/backend/server.py`
  - FastAPI 服务入口
- `app/backend/runtime_manager.py`
  - 任务生命周期管理
- `app/backend/config_store.py`
  - 配置读写
- `app/backend/event_bus.py`
  - 标准化事件推送
- `app/backend/artifact_service.py`
  - 中间产物和输出文件管理

### 9.2 需要从现有代码抽离的能力

需要把现在的这些能力从脚本风格改成库风格：

- 配置初始化
- 任务运行入口
- 日志注册
- 事件发射
- 阶段状态更新
- 任务结果汇总

理想目标是形成一个类似下面的接口：

```python
job_id = runtime.create_job(request)
runtime.start_job(job_id)
runtime.stream_events(job_id)
runtime.cancel_job(job_id)
runtime.get_artifacts(job_id)
```

## 10. 流式事件模型怎么设计

这是桌面端体验的关键。

目前仓库已经有日志文本，但 UI 真正需要的是结构化事件。

### 10.1 推荐事件类型

- `job_created`
- `phase_started`
- `phase_progress`
- `plan_created`
- `step_started`
- `thinking_summary`
- `tool_called`
- `tool_result`
- `artifact_created`
- `preview_ready`
- `warning`
- `error`
- `job_completed`
- `job_cancelled`

### 10.2 事件字段建议

```json
{
  "job_id": "job_001",
  "type": "tool_called",
  "phase": "phase3",
  "timestamp": "2026-03-23T10:00:00Z",
  "payload": {
    "tool_name": "cut_video",
    "summary": "裁剪第 2 个素材片段",
    "args_preview": {
      "start": 12.5,
      "end": 18.0
    }
  }
}
```

### 10.3 现有代码可复用的部分

当前 `graph.py` 已经有：

- `Plan`
- `Step`
- Phase 切换
- 工具调用轨迹日志

所以不是从零设计，而是把这些状态在运行时显式发出来。

## 13. 产物与数据管理设计

当前仓库已经有：

- `temp/`
- `user_temp/`
- `logs/`
- `memory_experience/`

桌面端应该把这些目录概念产品化，而不是直接暴露给用户。

### 13.1 建议的任务目录结构

```text
workspace/
  jobs/
    job_20260323_001/
      input/
      artifacts/
      output/
      logs/
      summary.json
      events.jsonl
```

### 13.2 为什么要按 Job 隔离

- 历史任务更清楚
- 便于恢复现场
- 便于展示任务详情
- 便于失败重跑
- 避免多个任务互相污染

### 13.3 当前代码里需要特别注意的点

现在很多工具默认围绕统一 `WORKSPACE` 工作，这在 CLI 模式下没问题，但桌面端后续要支持：

- 任务隔离
- 多任务排队
- 更稳定的缓存管理

因此后续需要逐步把“全局工作目录”改造成“Job 级工作目录”。

这是一个关键重构点。

### 14.1 原则

- 不让用户改源码
- 不把明文密钥写到仓库文件里
- UI 可以编辑配置

- 普通配置写 JSON
- API Key 存系统安全存储
- 启动时从安全存储读取并注入

### 14.3 连接测试能力

设置页必须支持：

- 测试文本模型
- 测试视频分析模型
- 测试 TTS 模型

并明确告诉用户：

- 哪个配置成功
- 哪个配置失败
- 失败原因是什么

这样才能让小白定位问题。

## 16. 为什么这个架构适合当前项目

因为当前项目的核心价值已经在 Python 里了：

- 工具链都在 Python
- Agent 编排都在 Python
- 日志和轨迹也在 Python

如果为了桌面端强行把核心搬到 Node/Rust，成本很高，收益很小。

正确做法是：

- Python 继续做引擎
- 前端负责体验
- 桌面壳负责分发和本地调用

这是一种最符合现状、风险最低、落地最快的方式。

## 17. 实施路线图

我建议分三期做。

### 17.1 Phase A：服务化重构

目标：

## 19. 关键风险与应对

### 19.1 风险：当前工具高度依赖全局工作目录

影响：

- 后续多任务和任务隔离会困难

应对：

- 第一版先单任务串行
- 第二版开始逐步改成 Job Workspace

### 19.2 风险：日志现在是“文本优先”，不是“事件优先”

影响：

- UI 解析成本高，不稳定

应对：

- 保留日志
- 新增结构化事件流

### 19.3 风险：一键安装复杂度高

影响：

- 可能拖慢整个项目节奏

应对：

- 先做“GUI + 本地服务”
- 再做“真正傻瓜式安装包”

### 19.4 风险：视频处理耗时长，用户会焦虑

影响：

- 产品观感差，用户会以为卡死

应对：

- 必须显示：
  - 当前阶段
  - 当前动作
  - 预计剩余时间
  - 中间产物
  - 最近成功动作

## 20. 成功标准

如果这个桌面版做对了，我认为应该满足以下标准：

- 用户不需要改源码
- 用户不需要手动在终端启动 Agent
- 用户输入 API 后可以直接用
- 执行过程不是黑盒
- 错误信息对普通人可理解
- 最终视频、日志、任务历史都能回看

---

## 21. 我对第一版的具体建议

如果现在立项，我建议第一版先不要追求“像 codex 一样什么都支持”，而是聚焦一个清晰版本：

### 第一版范围

- Windows 优先
- 单任务串行执行
- 本地桌面端
- 用户输入 API 后可直接跑
- 可上传本地素材
- 可看到三阶段流式过程
- 可预览最终视频
- 可查看任务历史和日志

### 第一版先不做

- 多任务并行
- 云同步
- 账号系统
- 插件市场
- 团队协作

这样更容易把核心体验做扎实。

## 22. 最推荐的落地路径

如果让我来主导，我会按这个顺序推进：

1. 先重构 Python 核心为可服务化调用
2. 再定义事件协议和任务数据模型
3. 再做 React 工作台 UI
4. 再接 Electron 成桌面端
5. 最后做打包、安装、依赖自检

这个顺序的好处是：

- 每一步都有独立产物
- 风险逐步暴露
- 不会一上来陷入桌面打包细节
- UI 和引擎能并行演进
